# X1 現地ハブWebアプリ UI計画

作成日: 2026-06-07

## UIの基本方針

HCD当日の主役は、来場者対応と付箋回収をしているメンバー。UIは「説明しなくても撮れる」「失敗してもすぐ分かる」「X1担当者が詰まりを直せる」を優先する。

このWebAppは2つの画面に分ける。

- スマホ用撮影画面: メンバーが自分のスマホで付箋を撮る
- X1管理画面: 受信状況、送信状況、エラー、再送を確認する

IP WebcamのようにX1で全シャッターを切る方式は、管理者用の予備操作として残す。主運用はスマホ側シャッター。

## 画面構成

```text
/
  メンバー用スマホ撮影画面

/admin
  X1管理画面

/admin/captures
  受信画像一覧

/admin/queue
  送信/OCRジョブ状態

/admin/settings
  イベント設定、送信先設定
```

初期実装では `/` と `/admin` だけで十分。細かい管理画面は後から分割する。

## スマホ用撮影画面

### 目的

メンバーが自分のスマホで付箋を撮影し、X1へ送る。

### 想定ユーザー

- Androidユーザー
- iPhoneユーザー
- 技術に詳しくないメンバー
- イベント中に短時間で操作するメンバー

### 画面遷移

```text
初期設定
  -> 撮影
  -> 確認
  -> 送信完了
  -> 次の撮影
```

### 初期設定画面

入力項目:

- 担当者名
- 担当場所
- デバイス名

担当場所の候補:

- 受付側
- 地図側
- 付箋回収
- 確認係
- その他

設計意図:

- 画像だけでは誰がどこで撮ったか分からないため、最低限の撮影元情報を残す
- 毎回入力させない。ブラウザのlocalStorageに保存する

### 撮影画面

主要要素:

- カメラプレビュー
- 大きいシャッターボタン
- ファイル選択ボタン
- 撮影済み件数
- X1接続状態

シャッターボタン:

- 画面下部に固定
- 親指で押しやすいサイズ
- 押した瞬間に状態が変わる
- 二重送信を防ぐ

ファイル選択ボタン:

- カメラAPIが使えない場合の保険
- iPhoneで権限やHTTPSに詰まった時の逃げ道

表示状態:

```text
待機中
撮影中
確認待ち
送信中
送信完了
送信失敗
```

### 確認画面

表示:

- 撮影画像プレビュー
- 送信ボタン
- 撮り直しボタン
- 補助入力

補助入力:

- ニックネーム
- ジャンル
- 地図エリア
- メモ

初期運用では補助入力は任意にする。撮影速度を落とさないため。

ジャンル入力は、可能ならシール色と合わせる。

- 黄色
- ピンク
- 水色
- 緑
- オレンジ
- 未選択

### 送信完了画面

表示:

- 送信成功
- 受付番号
- 次を撮るボタン
- 直近の送信履歴3件

受付番号例:

```text
HCD-0042
```

メンバーが「送れたか不安」にならないよう、番号をはっきり出す。

### エラー時

送信失敗時に出す内容:

- 何が起きたか
- 画像はスマホ側に保持されているか
- 再送ボタン
- ファイル選択に切り替えるボタン

文言例:

```text
X1に送信できませんでした。
画像はこの画面に残っています。Wi-Fiを確認して、再送してください。
```

## X1管理画面

### 目的

X1担当者が、当日の画像受信と送信/OCR処理の状態を一目で見る。

### レイアウト

デスクトップ前提。情報密度は高めにする。

上部:

- イベントID
- ローカル保存状態
- 受信件数
- 未送信件数
- エラー件数
- 研究室PC/クラウド接続状態

中央:

- 受信画像グリッド
- 最新画像を大きめに表示
- 各画像のステータスバッジ

右側:

- キュー状態
- エラーログ
- 再送操作

下部:

- 実行ログ
- 保存先パス
- 現在のWebApp URL

### ステータスバッジ

```text
captured
saved
queued
sent
ocr_running
pending_review
approved
upload_failed
ocr_failed
```

色の意味:

- 緑: 完了
- 青: 処理中
- 黄: 確認待ち
- 赤: 失敗
- 灰: 未処理

### 受信画像カード

各カードの表示:

- サムネイル
- 受付番号
- 撮影者
- 担当場所
- 撮影時刻
- ステータス
- 再送ボタン
- 詳細ボタン

カード内に長い文章は入れない。管理画面はスキャン性を優先する。

### 詳細モーダル

表示:

- 画像大表示
- record.json の要約
- ローカル保存パス
- 送信先URL
- エラー詳細
- 再送
- 除外

### 管理者用操作

- 全件再送
- 失敗分だけ再送
- manifest再生成
- ローカル保存フォルダを開く
- 設定再読み込み

「削除」は初期実装では置かない。誤操作で画像を失うのを避ける。

## デザイン方針

### フォント

Google Fonts の `Bitcount` を採用する。

Bitcountは表示用フォントとして使う。受付番号、画面タイトル、ステータス数値、短いラベルに使うと、イベント用の撮影端末らしい個性が出る。

ただし、Bitcountは日本語本文の可読性には向かないため、すべての文字に使わない。

使い分け:

- Bitcount
  - アプリ名
  - 受付番号
  - 件数カウンター
  - ステータスバッジ
  - 管理画面のメトリクス

- 日本語UI用フォント: DotGothic16
  - ボタン
  - 入力欄
  - 説明文
  - エラーメッセージ
  - 管理画面の一覧テキスト

Next.jsでは `next/font/google` で読み込む。

```ts
import { Bitcount, DotGothic16 } from "next/font/google";

export const bitcount = Bitcount({
  subsets: ["latin"],
  variable: "--font-bitcount",
});

export const dotGothic16 = DotGothic16({
  weight: "400",
  subsets: ["latin"],
  variable: "--font-dot-gothic",
});
```

CSS変数:

```css
:root {
  --font-display: var(--font-bitcount);
  --font-ui: var(--font-dot-gothic), "Yu Gothic", "Meiryo", system-ui, sans-serif;
}
```

実装時の注意:

- Bitcountは長文に使わない
- 日本語と英数字が混じる入力欄はUI用フォントにする
- 受付番号だけBitcountにして、本文と混ぜすぎない
- 数字は大きく使うと効果が出る
- イベント現場で読み間違いが起きないよう、重要な操作ボタンは装飾しすぎない
- DotGothic16も長文向きではないため、説明文は短く保つ

### スマホ画面

性格:

- 明るい
- 操作が少ない
- ボタンが大きい
- 屋外/イベント照明でも見やすい

構成:

- 1画面1目的
- 下部固定の主要操作
- 状態表示は短く
- 説明文を増やしすぎない

### X1管理画面

性格:

- 落ち着いた運用ダッシュボード
- 情報密度は高め
- 色は状態識別に使う
- 装飾より可読性

構成:

- ダーク寄りでもよい
- サムネイル一覧を中心にする
- エラーと未送信を目立たせる
- ログは折りたたみ可能にする

## APIとの関係

スマホ画面:

- `POST /api/captures`
  - 画像とメタデータを送る
- `GET /api/health`
  - X1との接続確認

管理画面:

- `GET /api/records`
  - 受信済み一覧
- `GET /api/queue`
  - 送信/OCRキュー状態
- `POST /api/sync/retry`
  - 再送
- `POST /api/manifest/rebuild`
  - manifest再生成

## UX上の重要判断

当日は「一発で綺麗に撮る」より「撮って送ったことが確実に分かる」ほうが大事。

そのため、UIでは以下を優先する。

- 送信完了番号を出す
- 失敗時に画像を消さない
- 再送できる
- X1管理画面で未送信を見つけられる
- スマホ操作は3ステップ以内にする

理想操作:

```text
開く
撮る
送る
```

またはカメラAPIが使えない場合:

```text
開く
画像を選ぶ
送る
```
